## 1. 引言与项目背景

### 1.1 校企合作课程的意义

本课程"人工智能工程实践"是学校与企业（AI创赢）合作开设的一门综合实践课程，旨在将前沿的AI技术与真实的软件开发场景相结合，培养学生在实际工程环境中运用AI工具独立解决问题的能力。

与传统以教师讲授为主的课程不同，本课程采用"项目驱动 + AI辅助"的教学模式：企业方提供真实业务需求与验收标准，学生以个人为单位独立完成完整的全栈项目交付。学生需要同时扮演需求分析师、后端开发、前端开发、测试工程师等多重角色，并在AI工具的辅助下高效完成各环节工作。

### 1.2 项目定位与核心价值

乌东文旅"衣食住行"综合服务平台的定位是：**构建乌东村一站式数字化文旅服务入口**。平台围绕贵州乌东村的苗族非遗文化资源，将非遗商品（衣）、餐饮美食（食）、民宿住宿（住）、线路门票（行）整合到一个统一的数字化系统中，辅以社区分享功能。

本项目的核心价值：

- **业务价值**：为乌东村搭建统一线上服务入口，帮助本地手工艺人、餐饮店主、民宿经营者实现数字化转型，推动乡村旅游经济发展
- **技术价值**：验证AI辅助全栈开发在真实项目中的可行性——本项目80%以上代码由Claude Code辅助生成，开发周期压缩至传统模式的1/3

## 3. 系统设计

### 3.1 系统架构

本项目采用**前后端分离**架构：

```
浏览器（React + Ant Design）
    ↕ HTTP/REST + Axios
Midway.js v3（Koa框架）
    ↕ TypeORM
MySQL 8.0
```

**[此处放置图片：系统架构图]**

### 3.2 技术选型

| 层级 | 技术 | 版本 | 选型理由 |
|---|---|---|---|
| 前端框架 | React | 18.x | 课程统一要求 |
| UI组件库 | Ant Design | 5.x | 企业级后台UI，表格/表单/图表丰富 |
| 图表库 | ECharts | 5.x | 数据看板需要折线图、柱状图、饼图 |
| HTTP客户端 | Axios | 1.x | 拦截器统一处理Token和错误 |
| 后端框架 | Midway.js | 3.x | 企业级Node.js框架，内置IoC、TypeORM集成 |
| ORM | TypeORM | 0.3.x | 实体管理、QueryBuilder灵活查询 |
| 数据库 | MySQL | 8.0 | 课程统一要求 |
| 认证 | JWT (jsonwebtoken) | 9.x | 无状态认证，适合前后端分离 |
| 密码加密 | bcryptjs | 3.x | 不可逆哈希，安全性高 |

### 4.2 管理员登录

**功能描述**：管理员使用用户名和密码登录管理后台，系统验证身份后返回JWT Token，前端存储到localStorage，后续请求自动携带Token。

**后端实现**：
- 接口：`POST /api/admin/login`
- 密码加密：使用bcryptjs进行哈希比对
- Token生成：JWT签名，24小时有效期
- 登录成功后自动更新 `last_login_time`

**前端实现**：
- 登录页（Login.tsx）：用户名 + 密码表单
- Token存储于 localStorage
- Axios请求拦截器自动附加 `Authorization: Bearer xxx`

**[此处放置图片：登录页面截图]**

### 4.3 管理员管理

**功能描述**：超级管理员可以对平台管理员账号进行增删改查操作。

- `GET /api/admin/list` — 分页列表（支持按用户名搜索、角色筛选、状态筛选）
- `GET /api/admin/detail` — 查看详情（含关联角色）
- `POST /api/admin/create` — 创建管理员（自动加密密码）
- `PUT /api/admin/update` — 更新管理员（支持修改密码）
- `DELETE /api/admin/delete` — 软删除（标记 is_deleted=1）
- `PUT /api/admin/status` — 切换启用/禁用状态

### 5.3 核心表设计示例

**管理员表（admin）**：
```sql
CREATE TABLE `admin` (
  `id` INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `username` VARCHAR(50) NOT NULL UNIQUE,
  `password` VARCHAR(255) NOT NULL,
  `real_name` VARCHAR(50),
  `email` VARCHAR(100),
  `role_id` INT UNSIGNED,
  `avatar` VARCHAR(255),
  `last_login_time` DATETIME,
  `status` TINYINT DEFAULT 1,
  `is_deleted` TINYINT DEFAULT 0,
  FOREIGN KEY (`role_id`) REFERENCES `role`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**订单表（order）**：
```sql
CREATE TABLE `order` (
  `id` INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `order_no` VARCHAR(50) NOT NULL,
  `user_id` INT UNSIGNED NOT NULL,
  `type` VARCHAR(20) NOT NULL COMMENT '商品/餐位/住宿/门票/路线',
  `status` TINYINT DEFAULT 0 COMMENT '0待支付 1已支付 2已确认 3已完成 4已取消',
  `amount` DECIMAL(10,2) DEFAULT 0,
  `merchant_id` INT UNSIGNED,
  `is_deleted` TINYINT DEFAULT 0
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**所有表统一设计规范**：
- 主键使用 `INT UNSIGNED AUTO_INCREMENT`
- 软删除使用 `is_deleted TINYINT DEFAULT 0`
- 时间戳使用 `created_at` / `updated_at`（`DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`）
- 字符集统一 `utf8mb4`

## 8. 技术难点与解决方案

### 8.1 TypeORM 实体字段映射问题

**问题**：实体属性使用驼峰命名（如 `isDeleted`），数据库字段使用下划线命名（如 `is_deleted`）。使用 `Repository.findAndCount({ where: { isDeleted: 0 } })` 时，TypeORM 未能正确将驼峰属性名映射为数据库列名，导致查询条件不生效，返回空数据。

**解决方案**：所有查询改用 `createQueryBuilder`，在 `.where()` 和 `.andWhere()` 中直接使用数据库实际列名（snake_case）：
```typescript
this.adminModel.createQueryBuilder('a')
  .where('a.is_deleted = 0')
  .andWhere('a.status = :status', { status: 1 })
  .getManyAndCount();
```

### 8.2 CORS 预检请求 404

**问题**：浏览器在跨域请求前会发送 OPTIONS 预检请求。使用 `this.app.useMiddleware()` 注册的 CORS 中间件排在 Midway 内置路由之后，OPTIONS 请求先被路由匹配 → 没有对应的 `@Options` 路由 → 返回 404，CORS 中间件根本没有执行。

**解决方案**：CORS 必须通过 Koa 原生 `app.use()` 注册在整个中间件链的最前端：
```typescript
this.app.use(async (ctx, next) => {
  ctx.set('Access-Control-Allow-Origin', '*');
  // ...
  if (ctx.method === 'OPTIONS') { ctx.status = 204; return; }
  await next();
});
```

### 8.3 MySQL ONLY_FULL_GROUP_BY 错误

**问题**：Dashboard 统计接口中，`SELECT` 使用了 `DATE_FORMAT()` 函数，但 `GROUP BY` 使用的是 `DATE()` 函数，两者表达式不一致。MySQL 的 `ONLY_FULL_GROUP_BY` 模式要求 SELECT 中的非聚合表达式必须出现在 GROUP BY 中。

**解决方案**：统一 SELECT 和 GROUP BY 中的表达式，均使用 `DATE(created_at)`：
```sql
SELECT DATE(o.created_at) as date, COUNT(*) as count
FROM `order` o
GROUP BY DATE(o.created_at)
ORDER BY date ASC
```

### 8.4 时区导致日期查询为空

**问题**：Node.js 的 `new Date()` 传给 MySQL 时被转换为 UTC 时间字符串，与数据库中的 `DATETIME` 字段（本地时间）对比不一致，导致"今日数据"统计始终为 0。

**解决方案**：使用纯日期字符串比较，绕开时区转换：
```typescript
function localDate(d: Date): string {
  return `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`;
}
// SQL: WHERE DATE(o.created_at) = '2026-06-28'
```

### 8.5 bcryptjs 密码哈希不兼容

**问题**：管理员密码哈希由外部工具生成，`bcryptjs.compareSync()` 验证始终返回 false。

**解决方案**：密码的生成和验证必须使用同一个库：
```bash
node -e "const bcrypt = require('bcryptjs'); console.log(bcrypt.hashSync('123456', 10));"
```
将生成的 `$2a$10$...` 哈希写入数据库后，`bcryptjs.compareSync('123456', hash)` 返回 true。

### 8.6 Ant Design Menu 类型错误

**问题**：`{ type: 'divider' }` 在 Ant Design 5/6 的 `items` 属性中类型不兼容（TS2322）。

**解决方案**：使用 `as const` 窄化字面量类型，或使用 `MenuProps['items']` 显式声明类型：
```typescript
const menuItems: MenuProps['items'] = [
  // ...
  { type: 'divider' } as const,
  { type: 'group', label: '分组标题', children: [...] },
];
```

### 8.7 前端数据解析多取一层

**问题**：多个页面出现"后端有数据但前端显示空/0"的问题。原因是 Axios 响应拦截器已经 `response.data` 取过一次（`{ success, data, total }`），页面代码又取了 `res.data?.data`，导致实际取到 undefined。

**解决方案**：统一规范——`data` 字段已在拦截器层取出，页面直接使用 `res.data`。

### 8.8 中文乱码

**问题**：部分页面中文显示为乱码。

**解决方案**：三层修复——① TypeORM 连接配置 `charset: 'utf8mb4'`；② mysql2 驱动 `extra.charset: 'utf8mb4'`；③ 中间件设置 `Content-Type: application/json; charset=utf-8`。部分源文件编码损坏的直接重写文件。

## 9. AI辅助开发体会

### 9.1 AI工具使用情况

本项目使用 **Claude Code** 作为核心AI开发工具，在以下环节深度应用：

| 环节 | AI 参与度 | 说明 |
|---|---|---|
| 需求分析 | 30% | AI辅助解读需求文档，提取功能清单 |
| 数据库设计 | 70% | AI生成建表SQL和种子数据 |
| 后端代码 | 85% | Entity/Service/Controller 几乎全由AI生成 |
| 前端代码 | 80% | 页面组件和API封装由AI生成 |
| Bug排查 | 90% | AI分析错误日志并给出修复方案 |
| 文档撰写 | 60% | AI辅助撰写报告 |

**总体代码产出中，约80%由AI辅助生成。**

### 9.2 AI的优势

- **模板化代码效率极高**：CRUD、分页列表、表单页面对AI来说已有固定模式，生成速度快且结构一致
- **跨文件一致性**：AI能记住项目风格（如QueryBuilder写法、响应格式），新代码自然融入
- **Bug排查精准**：报错信息 + 代码上下文 → AI通常能1轮定位根因
- **全栈无缝切换**：同一对话中从数据库→后端→前端全部覆盖，不存在"前端等后端"的协作延迟

### 9.3 AI的局限性

- **版本兼容性问题**：AI生成的 TypeORM v1.0 代码不兼容本项目的 TypeORM v0.3，需要人工验证和修正
- **时区/编码等环境问题**：AI默认生成通用代码，对 `ONLY_FULL_GROUP_BY`、UTC偏移等具体环境细节需要人工排查
- **UI细节**：图标名称、Menu类型定义等小问题容易出错，需要反复调试
- **需求理解**：AI不能替代人工对需求文档的深度阅读和判断

### 9.4 开发模式总结

**"AI生成 → 人工审阅 → 编译验证 → 功能测试 → Bug修复"** 是本项目的高效迭代模式。

相比传统开发，AI辅助开发大幅降低了"从零到一"的启动成本——整个项目从空目录到可运行的全栈系统，核心开发时间约8小时。但AI生成的代码仍需人工验证每一步的正确性，特别是版本兼容性、环境配置和边界条件。
