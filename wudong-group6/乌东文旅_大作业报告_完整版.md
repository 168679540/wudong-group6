## 1. 引言与项目背景

### 1.1 校企合作课程的意义

本课程"人工智能工程实践"是学校与企业（AI创赢）合作开设的一门综合实践课程，旨在将前沿的AI技术与真实的软件开发场景相结合，培养学生在实际工程环境中运用AI工具独立解决问题的能力。

与传统以教师讲授为主的课程不同，本课程采用"项目驱动 + AI辅助"的教学模式：企业方提供真实业务需求与验收标准，学生以个人为单位独立完成完整的全栈项目交付。学生需要同时扮演需求分析师、后端开发、前端开发、测试工程师等多重角色，并在AI工具的辅助下高效完成各环节工作。

### 1.2 项目定位与核心价值

乌东文旅"衣食住行"综合服务平台的定位是：**构建乌东村一站式数字化文旅服务入口**。平台围绕贵州乌东村的苗族非遗文化资源，将非遗商品（衣）、餐饮美食（食）、民宿住宿（住）、线路门票（行）、社区游记分享整合到一个统一的数字化系统中。

乌东村位于贵州省黔东南苗族侗族自治州，是苗族文化保存最为完好的古村落之一，拥有苗族银饰锻造、蜡染刺绣、苗家长桌宴、特色民宿、苗寨梯田与节庆文化等丰富的文旅资源。然而，现有运营以线下为主，缺乏统一的线上服务入口。本项目正是为了解决这一痛点：通过构建数字化平台，让游客能够一站式完成浏览、下单、预订、评论、分享的全流程，同时为本地商家提供店铺管理、订单管理的数字化工具。

本项目的核心价值：

- **业务价值**：为乌东村搭建统一线上服务入口，帮助本地手工艺人、餐饮店主、民宿经营者实现数字化转型，推动乡村旅游经济发展；为游客提供便捷的一站式线上服务体验
- **技术价值**：验证AI辅助三端全栈开发在真实项目中的可行性。本项目80%以上代码由Claude Code辅助生成，开发周期压缩至传统模式的1/3
- **教学价值**：完整覆盖需求分析、数据库设计、后端开发、前端开发、小程序开发、测试验证的全生命周期，是对软件工程实践能力的全面训练

## 3. 系统设计

### 3.1 系统架构

本项目采用**前后端分离 + 小程序独立开发**的三端架构。后端通过 Midway.js 提供统一的 RESTful API，PC 端和管理后台共用同一套 React 项目（通过路由区分），微信小程序使用原生框架独立开发。

架构的核心设计原则是：**一套后端 API 服务三端**。无论是 PC 端、管理后台还是小程序，均通过 HTTP 请求访问同一组 API 接口，共享同一份数据和业务逻辑。这保证了数据一致性，避免了业务规则的重复实现。

│  │   PC 网页端          │  │   管理后台 Web     │  │ 小程序  │ │
│  │   React 19           │  │   React 19        │  │ 原生框架 │ │

### 3.4 安全设计

| 机制 | 实现方式 |
|------|---------|
| **身份认证** | JWT Token，登录后返回，PC 端存储于 localStorage，小程序存储于 storage。请求头携带 `Authorization: Bearer xxx` |
| **鉴权控制** | Auth 中间件 + 白名单机制（WHITE_LIST 数组），公开页面 API 无需 Token，管理操作需登录验证。白名单当前包含约 40 个公开路径 |
| **密码安全** | bcryptjs 10 轮哈希存储，不可逆，防止数据泄露后被解密 |
| **CORS 防护** | Koa 原生 `app.use()` 注册 CORS 中间件在中间件链最前端，配置允许的来源和方法，拦截 OPTIONS 预检请求 |
| **操作审计** | 全部管理操作（POST/PUT/DELETE）通过 OperationLog 中间件自动记录到 operation_log 表，包含操作人、操作类型、目标路径、请求参数、IP 地址 |
| **软删除** | 所有业务表统一使用 `is_deleted` 字段，删除操作仅标记而非物理删除，支持数据恢复 |

### 4.2 模块一：衣·非遗商品

衣模块是整个平台上功能最完善的模块，实现了从商品浏览、筛选、详情查看、规格选择、加入购物车、立即购买到评价反馈的完整购物闭环。

#### 4.2.1 PC 端功能详情

PC 端衣模块页面路由为 `/pc/product`，是游客访问最频繁的页面。

**分类与搜索**：页面顶部提供了动态分类 Tag 选择（全部/银饰/蜡染/刺绣/服饰/其他），分类列表从 `product_category` 表动态加载，管理员在后台新增的分类会自动出现在 PC 端。同时支持关键词搜索、价格区间筛选（最低价/最高价 InputNumber）、评分筛选（Rate 组件，选择最低评分过滤）和排序（按销量/价格从低到高/价格从高到低）。

**商品卡片**：每个商品以 Ant Design Card 组件展示，包含 220px 高度封面图、商品名称、分类标签、价格（红色加粗显示）、运费标签（橙色 Tags）、销量信息。卡片左上角有心形收藏按钮（HeartOutlined / HeartFilled 切换），右上角显示"热销"Tag（销量 > 50）。

**详情弹窗**：点击"查看详情"打开 720px 宽的 Modal 弹窗。弹窗内展示：商品大图（400px 高度）、商品名称、分类标签、价格、库存、销量、商品描述、退票政策。弹窗下方展示商品评价列表（包含评分星星、评价文字、商家回复、追评内容），同时提供"写评价"输入框供游客提交评价。弹窗底部提供"返回"和"立即购买"两个操作按钮。

**规格选择弹窗**：点击"立即购买"弹出 480px 宽的规格选择 Modal。弹窗内展示：商品图片、商品名称、尺寸下拉选择（读取 product.specs JSON 字段的"尺寸"数组）、颜色下拉选择（读取"颜色"数组）、数量输入（InputNumber，最大值为商品库存）、实时总价计算（商品金额 + 运费 = 合计）。弹窗底部提供"加入购物车"和"立即购买"两个按钮。

**购物车系统**：购物车通过 React Context（CartContext）实现全局状态管理，通过 localStorage 持久化。页面右上角显示购物车图标 🛒 和数量角标。点击购物车图标打开右侧 Drawer 抽屉面板，展示购物车内所有商品：商品图片、名称、规格标签（如"尺寸:中号 颜色:蓝色"）、单价、数量输入（InputNumber，可修改）、删除按钮。面板底部显示合计金额和"清空"/"去结算"按钮。点击"去结算"时，购物车会逐件调用 createOrder API 批量创建订单，创建前校验库存和商品状态，失效商品红色标记提示。

### 4.8 统一购物车系统

购物车是一个跨模块的公共功能。PC 端通过 React Context（CartContext）实现全局状态管理，通过 localStorage 持久化。小程序端通过 wx.setStorageSync 存储。

**PC 端实现细节**：

- CartProvider 组件包裹整个应用（在 App.tsx 中），通过 React Context 向所有子组件提供购物车状态
- 每个商品在购物车中的唯一标识 uid 由 `productId + 规格组合` 生成（如 `p1_中号_蓝色`），确保同商品不同规格作为不同条目
- 购物车项包含字段：uid、productId、name、image、price、quantity、specs（规格对象）、merchantId
- CartDrawer 组件从 Context 读取 items，通过 Ant Design Drawer 右侧滑出展示
- 结算流程：点击"去结算" → 逐件调用 createOrder API → 成功后清空购物车并关闭面板
- 库存校验：结算前逐一检查商品详情（调用 getProductDetail），校验商品状态（下架/库存不足）并标红提示
- 初始数据从 localStorage 恢复，每次更新同步写入 localStorage

**小程序端实现细节**：

- 购物车数据存储在 `wx.wudong_cart` 中，通过 wx.getStorageSync/wx.setStorageSync 读写
- 购物车页面展示每件商品的图片、名称、价格、数量（可输入修改）、删除按钮
- 底部固定条显示合计金额和结算按钮
- 结算流程：遍历购物车逐件调用 createOrder → 成功后 toast 提示并清空购物车

### 4.11 时空体系与数据一致性

**时区处理**：MySQL 默认 UTC 时区。后端通过配置 `timezone: '+08:00'` 和 SQL 中 `CONVERT_TZ(o.created_at, '+00:00', '+08:00')` 函数确保统计数据以北京时间汇总。

**软删除机制**：所有业务表统一使用 `is_deleted TINYINT DEFAULT 0` 字段，删除操作仅标记 `is_deleted = 1`。所有查询 where 条件均包含 `is_deleted = 0`。

## 8. 技术难点与解决方案

### 8.1 TypeORM 实体字段映射问题

**问题**：实体使用驼峰命名（`isDeleted`），数据库使用下划线命名（`is_deleted`）。TypeORM 在 `findAndCount({ where: { isDeleted: 0 } })` 时未能正确将驼峰属性名映射为数据库列名，导致查询条件不生效。

**解决方案**：所有查询改用 `createQueryBuilder`，在 `.where()` 中直接使用数据库实际列名。例如：`this.model.createQueryBuilder('a').where('a.is_deleted = 0')`。这种写法虽然更冗长，但能确保查询条件与数据库列名完全一致。

### 8.2 CORS 预检请求 404

**问题**：浏览器在跨域请求前发送 OPTIONS 预检请求。通过 Midway 的 `this.app.useMiddleware()` 注册的 CORS 中间件排在路由之后，OPTIONS 请求先被路由匹配，但没有对应的 `@Options` 路由，返回 404，CORS 中间件根本没执行。

**解决方案**：CORS 通过 Koa 原生 `this.app.use()` 注册在整个中间件链的最前端，拦截 OPTIONS 请求直接返回 204：`if (ctx.method === 'OPTIONS') { ctx.status = 204; return; }`。

### 8.3 MySQL 时区导致今日数据为 0

**问题**：MySQL 默认 UTC 时区（慢 8 小时），加上 Node.js `new Date()` 传给 MySQL 后的转换差异，导致数据库中的"今日数据"统计始终为 0。例如，北京时间 7 月 1 日 14 点创建的订单，在 MySQL 中存储为 6 月 30 日 06:00 UTC。

**解决方案**：两步修复——① TypeORM 连接配置 `timezone: '+08:00'`；② 在 Dashboard 统计 SQL 中使用 `CONVERT_TZ(o.created_at, '+00:00', '+08:00')` 确保所有查询基于北京时间比较。

### 8.4 小程序图片路径不显示

**问题**：后端 API 返回的商品图片路径为相对路径（如 `/images/product1.jpg`），在小程序端直接使用 wxml `<image src="/images/product1.jpg">` 时，小程序无法识别相对路径。

**解决方案**：在小程序每个页面的数据加载成功后，统一调用 `fixImg()` 函数将相对路径补全为完整 URL（`http://127.0.0.1:3000/images/product1.jpg`）。该函数检查 URL 是否以 `/` 开头，是则拼上域名前缀。

### 8.5 小程序 Picker 与弹窗 mask 冲突

**问题**：Picker 选择器选择完毕后，确认事件会触发底层 mask 的 tap 事件，而 mask 的 `catchtap="closeXxx"` 会立即关闭弹窗。导致用户选完一个规格后弹窗就关闭了，无法完成完整的规格选择。

**解决方案**：去掉 mask 层的 `catchtap` 关闭事件，改为只在弹窗内的关闭按钮上绑定关闭逻辑。用户必须点击"取消"按钮才能关闭弹窗，Picker 的选择不受干扰。

### 8.6 ECharts 图表数据不更新

**问题**：首次渲染时图表数据为空，后续 API 数据返回后 `setOption` 无法正确更新图表，数据始终显示为 0。根因是组件初始化时 DOM 未完全附着。

**解决方案**：重写 ECharts 组件：初始化和销毁各一次（`useEffect with []`），数据更新时仅调用 `chartInstance.setOption(option, true)` 复用已有实例。同时在图表父组件通过 `key={数据连接字符串}` 确保 option 对象引用变化时强制重建图表。

### 8.7 评价提交后不显示

**问题**：用户在 PC 端或小程序提交评价成功，但评价列表不显示。根因是评价创建 service 的 `create()` 方法中未显式设置 `status = 1`，而列表查询条件包含 `status = 1`，导致新评价虽然存入数据库但被过滤。

**解决方案**：在所有评价 Service（product-review/restaurant-review/homestay-review/ticket-review）的 `create()` 方法中显式设置 `r.status = 1`。

### 8.8 评价模块数据混杂

**问题**：商品评价管理页面显示了餐厅的评价，民宿评价管理页面显示了票务评价。根因是 `product_review` 表共用，但 adminList 查询只加了 `is_deleted = 0` 条件，没有用关联 ID 过滤。

**解决方案**：在商品评价 adminList 中添加 `r.product_id IS NOT NULL`，餐厅评价添加 `r.restaurant_id IS NOT NULL`，民宿评价添加 `r.homestay_id IS NOT NULL`，票务评价添加 `r.ticket_id IS NOT NULL`。

### 8.9 Entity 列映射缺失

**问题**：ProductReview 实体中缺少 `restaurantId`、`homestayId`、`ticketId` 三个字段的 `@Column` 映射。TypeORM 保存评价对象时，即使 JS 对象上有这些属性，也会因为实体类没有对应映射而丢弃，导致评价的关联 ID 为 NULL，查询列表时过滤条件匹配不到。

**解决方案**：在 ProductReview 实体中显式添加三个字段的 `@Column` 装饰器映射，并设为 `nullable: true`，与数据库列定义和实体查询逻辑保持一致。

### 9.2 AI 的优势

- **效率提升巨大**：CRUD、分页列表、表单页面等模板化代码，AI 生成速度远超人工编写
- **跨文件一致性**：AI 记住项目风格，新代码自然融入
- **全栈无缝切换**：同一对话中覆盖数据库→后端→PC前端→小程序
- **Bug 排查精准**：报错信息 + 代码上下文 → AI 通常能 1-2 轮定位根因

### 9.3 AI 的局限性

- **版本兼容性**：AI 生成的 API 引用可能不匹配实际安装的版本
- **环境细节**：时区、编码、MySQL 模式等特定环境配置需人工调试
- **UI 细节**：图标名称拼写、Menu 类型不兼容等小问题需要多次调试

### 9.4 开发流程总结

本项目的有效迭代模式：**"需求分析 → AI 生成代码 → 编译验证 → 功能测试 → Bug 修复 → 截图验证"**。

### 11.3 不足与展望

| 不足 | 改进方向 |
|------|---------|
| 图片存储在本地的相对路径 | 接入 OSS 对象存储，支持文件直接上传 |
| 支付为模拟流程 | 接入微信支付 SDK，实现在线支付 |
| 无单元测试覆盖 | 使用 Jest 补充 Service 层单元测试 |
| 暂无 Docker 部署 | 编写 Docker Compose 实现一键部署 |
| 缺少 WebSocket 实时推送 | 订单状态变更、审核结果可实时推送 |
| 图片内容审核 | 接入 AI 图片识别服务自动审核上传图片 |
